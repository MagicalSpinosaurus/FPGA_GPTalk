너는 Zybo Z7-20 / Zynq-7020에서 SmolLM2 추론을 구현하는 프로젝트의 코딩 보조자다.

이 전역 지침은 프로젝트 전체에서 최우선으로 따른다. 단계별 프롬프트나 기존 문서가 이 지침과 충돌하면 이 전역 지침을 우선한다.

프로젝트 목표:

* SmolLM2-135M-Instruct를 우선 대상으로 한다.
* 기본 모델 산출물은 이미 양자화된 SmolLM2-135M-Instruct Q8_0 GGUF이다.
* 기본 weight 경로는 GGUF-first Q8_0 경로다.
* PC에서 GGUF를 inspect하고 Q8_0 weight/scale을 FPGA layout과 runtime용 DMA/stream packet으로 변환한다.
* BF16 원본에서 직접 INT8로 양자화하는 경로는 기본 경로가 아니라 fallback이다.
* 기본 구현과 문서는 먼저 GGUF Q8_0을 기준으로 작성한다.
* PS Linux가 tokenizer, 파일 IO, 전체 layer scheduling, RMSNorm, RoPE, SiLU/SwiGLU, softmax, sampling, KV cache, HDMI/USB console chat을 담당한다.
* PL FPGA는 Transformer 내부 Linear/GEMV를 가속한다.
* 최종 목표는 SmolLM2 runtime의 모든 Q8_0 GEMV/Linear 호출이 FPGA backend를 통과하는 것이다.
* 처음부터 Transformer 전체를 Verilog로 만들지 않는다.
* 처음부터 custom AXI master를 직접 만들지 않는다.
* 본편은 AXI-Lite control/status + AXI DMA + AXI-Stream GEMV IP 구조를 사용한다.
* AXI DMA IP의 MM2S/S2MM AXI master 경로는 사용한다. 금지하는 것은 직접 custom AXI master RTL을 새로 만드는 것이다.

Q8_0 수치 정책:

* Q8_0은 32개 weight마다 fp16 scale이 하나 있다.
* GGUF의 fp16 scale은 PC 변환기에서 fixed-point scale_q로 변환한다.
* 기본 SCALE_SHIFT는 20으로 둔다.
* FPGA 본 경로 mode=0은 scaled row output이다.
* mode=0에서 FPGA는 int16 activation × int8 weight MAC으로 block_acc_i32를 만들고, block_acc_i32 × scale_q를 fixed-point로 적용한 뒤 row별 scaled_output_i32를 출력한다.
* CPU runtime은 mode=0 scaled_output_i32를 float32 runtime으로 이어 받아 후속 연산을 수행한다.
* mode=1은 debug 전용 block_acc_i32[row][block] 출력이다.
* block_acc_i32는 Q8_0의 최종 출력이 아니며, scale/weight alignment를 추적하기 위한 디버그 값이다.
* RTL/testbench/golden 명세에서는 block index, lane order, scale_q order, packet order를 항상 명시한다.
* Verilog에서 real, shortreal, floating-point operator IP를 사용하지 않는다.
* FPGA 내부에서는 fixed-point scale_q만 사용한다.

데이터 이동 정책:

* AXI-Lite는 control/status/version/start/done/error/register map 용도로만 사용한다.
* AXI-Lite로 weight_i8, scale_q, input_i16, output_i32 전체를 반복 전송하는 구조는 최종 경로로 금지한다.
* weight_i8 + scale_q packet은 AXI DMA MM2S -> AXI4-Stream FIFO -> GEMV s_axis로 공급한다.
* GEMV output은 AXI DMA S2MM 또는 BRAM/AXI BRAM Controller 같은 burst 가능한 경로로 CPU에 전달한다.
* input activation vector는 BRAM, DMA packet, 또는 명확한 burst 가능한 경로를 사용한다.
* INPUT_DATA, STREAM_DATA, RESULT_DATA 같은 단일 AXI-Lite data register 반복 전송 방식은 deprecated/debug-only로만 허용한다.
* smoke bitstream은 version/status register 확인용일 뿐, full GEMV datapath 성공으로 취급하지 않는다.

하드웨어 구조 정책:

* `vivado_ip/rtl/gemv_q8_0_stream_core.v` 같은 AXI-Stream 계산 core는 보존한다.
* AXI-Lite data pump wrapper는 최종 구조에서 제거하거나 debug-only로 격리한다.
* Vivado block design에는 AXI DMA, AXIS FIFO, PS HP DDR 경로가 있어야 한다.
* GPTalk.xpr 내부 BD에 PS7 + AXI DMA + AXIS FIFO + GEMV DMA top + HP0 경로를 만든다.
* FCLK_CLK0는 route/timing 안정화를 위해 처음에는 50 MHz로 둘 수 있다.
* PL clock을 낮춰도 PS CPU clock을 낮추는 것은 아니다.
* 50 MHz route 성공 후 75 MHz, 100 MHz 순서로 올린다.
* 기능 축소로 timing을 해결하지 않는다.
* q_proj 하나만 처리하는 fake accelerator, fake_gemv 하드코딩 IP, scale 제거, mode 제거, CPU scale fallback은 금지한다.
* 모든 GEMV를 FPGA backend로 보내는 목표는 유지하되, 내부 구현은 하나의 16-lane GEMV engine을 row-by-row 또는 tile-by-tile로 시간분할 재사용해도 된다.

RTL 코딩 원칙:

* 한 번에 큰 코드를 만들지 말고, 작은 모듈 단위로 작성한다.
* 합성 가능한 Verilog/SystemVerilog를 작성한다.
* 시뮬레이션 전용 문법은 testbench에만 사용한다.
* 클럭 always 블록에서는 nonblocking assignment(<=)를 사용한다.
* ready/valid handshake는 valid를 소비될 때까지 유지하는 방식으로 작성한다.
* 큰 buffer는 FF/LUT array가 아니라 BRAM/XPM memory로 유도한다.
* 16-lane MAC, scale multiply, round shift, row accumulation을 한 사이클 조합 경로에 몰아넣지 말고 pipeline stage로 분리한다.
* 권장 stage:
  - Stage A: stream packet 수신, scale_q latch, weight_i8 unpack
  - Stage B: input_i16 BRAM read
  - Stage C: 16-lane multiply
  - Stage D: partial sum adder tree
  - Stage E: block_acc_i32 latch
  - Stage F: block_acc_i32 × scale_q multiply
  - Stage G: rounding shift
  - Stage H: row_acc accumulation
  - Stage I: output_i32 saturation/writeback
* 불필요한 keep, dont_touch, mark_debug로 최적화를 막지 않는다.
* 인터페이스를 바꿔야 할 경우 먼저 이유와 변경 전/후 포트 목록을 설명한다.

Vivado implementation timing/performance 최적화 정책:

* full GEMV datapath bitstream을 만들 때 Vivado implementation run을 기본값 그대로 두지 않는다.
* 첫 목표 clock은 PL 50 MHz로 두되, implementation은 timing/performance-oriented strategy/directive를 사용한다.
* 현재 Vivado 2024.2.2에서 지원되는 strategy/directive를 먼저 확인한다.
* 가능한 경우 implementation strategy를 Performance_Explore 또는 timing closure에 유리한 strategy로 설정한다.
* opt_design, place_design, phys_opt_design, route_design에는 Explore/AggressiveExplore 계열 directive를 우선 시도한다.
* post-route phys_opt가 지원되면 활성화한다.
* 적용한 strategy/directive를 `logs/vivado_impl_strategy.txt`에 기록한다.
* 지원되지 않는 directive 이름을 무리하게 쓰지 말고, Vivado가 지원하는 가장 가까운 timing/performance directive로 fallback한다.
* directive fallback이 발생하면 어떤 directive가 실패했고 무엇으로 대체했는지 문서화한다.
* timing/performance directive 사용은 필수지만, 기능 축소로 성공 처리하면 안 된다.
* lane 수 축소, scale 제거, mode 제거, fake_gemv 하드코딩, AXI-Lite 데이터펌프 유지 같은 방식으로 timing을 맞추면 실패로 본다.
* timing 실패 시 WNS/TNS, top failing paths, congestion report를 보고 RTL pipeline/BRAM/stream 구조를 수정한다.

권장 Tcl 방향:

```tcl
# 실제 지원 여부 확인 후 적용한다.
set_property strategy Performance_Explore [get_runs impl_1]

set_property STEPS.PLACE_DESIGN.ARGS.DIRECTIVE Explore [get_runs impl_1]
set_property STEPS.PHYS_OPT_DESIGN.IS_ENABLED true [get_runs impl_1]
set_property STEPS.PHYS_OPT_DESIGN.ARGS.DIRECTIVE AggressiveExplore [get_runs impl_1]
set_property STEPS.ROUTE_DESIGN.ARGS.DIRECTIVE Explore [get_runs impl_1]
```
